## 引言

工程界有句老话——"缓存统治一切"（Cache Rules Everything Around Me），对于 AI Agent 来说，这条铁律同样成立。

像 Claude Code 这样长时间运行的 Agent 产品，之所以在经济上可行，完全建立在提示词缓存（Prompt Caching）之上。它让我们能够复用之前轮次的推理计算，大幅降低延迟和成本。

在 Claude Code 团队，我们围绕提示词缓存构建了整个运行时框架。高缓存命中率直接拉低使用成本，让我们有能力为订阅用户提供更宽松的调用额度——这并非锦上添花，而是产品的生命线。因此，我们会像监控系统可用性一样监控缓存命中率：一旦命中率过低，立即触发告警，甚至直接升级为 SEV（严重事件）。

以下是我们在规模化优化提示词缓存过程中，积累下来的经验教训——其中不少结论相当反直觉。

---

## 提示词缓存的工作原理

提示词缓存采用**前缀匹配**机制——API 会缓存从请求开头一直到每个 `cache_control` 断点之间的全部内容。这意味着，提示词中内容的排列顺序，几乎决定了一切。

如果前缀中的任意一个字节发生了变化，该字节之后的**所有**缓存都会失效，必须重新计算。这是一种严格逐字节匹配的前缀比对——从请求的第一个字节开始，差之毫厘，失之千里。

正是这个硬约束，塑造了 Claude Code 的几乎每一项架构决策。

---

## 最优的提示词布局

Claude Code 在每个 API 请求中，都遵循同一个原则：**静态内容放在最前面，动态内容放在最后面**。这能最大化不同请求之间共享缓存前缀的概率。

```
┌─────────────────────────────────────────────┐
│ 1. 静态系统提示词 + 全部工具定义              │  ← 全局缓存（所有会话、所有用户共享）
├─────────────────────────────────────────────┤
│ 2. CLAUDE.md（项目配置）                     │  ← 项目级缓存
├─────────────────────────────────────────────┤
│ 3. 会话上下文（环境信息）                     │  ← 会话级缓存
├─────────────────────────────────────────────┤
│ 4. 对话消息（用户/Agent/工具结果）            │  ← 每轮追加（动态层，在此设置缓存断点）
└─────────────────────────────────────────────┘
```

**第一层 —— 系统提示词 + 工具定义（全局缓存）：** 包含核心 Agent 指令、全部工具定义（schema）和输出风格设定。缓存 TTL 为 1 小时。升级 Claude Code、修改工具定义、调整内置工具的 deny 规则，都会导致这一层缓存失效。

**第二层 —— CLAUDE.md（项目级缓存）：** 来自目录层级中的 `CLAUDE.md` 文件、自动记忆（`MEMORY.md` 前 200 行）以及无作用域规则。这些内容在会话启动时一次性加载——会话中途对 CLAUDE.md 的修改，需要 `/clear`、`/compact` 或重启会话才能生效。

**第三层 —— 会话上下文（会话级缓存）：** Git 状态快照、工作目录、操作系统与 Shell 版本、自动记忆路径等。这些信息被嵌入到系统提示词中，因此不同目录下的会话无法共享缓存——前缀从一开始就分叉了。

**第四层 —— 对话消息（动态层）：** 用户消息、Claude 的回复、工具执行结果以及 `<system-reminder>` 标签。这是唯一每轮都会增长的一层。所有动态内容都放在这里，以避免破坏前缀缓存。

`cache_control` 断点设置在系统提示词块之后（TTL 1 小时）和最后一条用户消息之后（TTL 5 分钟）。API 要求 1 小时 TTL 的内容必须位于 5 分钟 TTL 内容之前——这一顺序不可颠倒。

## 六条核心经验

### 经验一：为缓存而设计提示词布局

提示词中内容的排列顺序，是影响缓存命中率的头号因素。**静态内容必须在前，动态内容必须在后。**

这条原则看似简单，实际上极其脆弱。以下是实践中真实踩过的坑：

- 在静态系统提示词里嵌入精确时间戳（每次请求都变，从源头摧毁整个缓存链）
- 以非确定性顺序排列工具定义（常见于从 Set/HashMap 等无序数据结构生成工具列表时）
- 在开发过程中更改工具参数（比如调整 `AgentTool` 可以调用哪些 Agent）
- 切换推理强度等级（每个 effort level 拥有独立的缓存键）
- 开启快速模式（会添加一个新的请求头，成为缓存键的一部分）

一个实用的经验法则：**如果某段内容会在请求之间变化，就把它尽可能推到提示词的末尾。**

---

### 经验二：用消息传递更新，而非修改提示词

当提示词中的信息过时了——时间变了、文件被外部编辑了、工作目录切换了——**永远不要直接修改提示词本身**。那会导致缓存全量失效。

正确做法是：通过 Agent 下一轮对话中的**消息**来注入更新信息。Claude Code 的做法是在下一条用户消息或工具结果中插入 `<system-reminder>` 标签：

> "在 Claude Code 中，我们会在下一条用户消息或工具结果里添加一个 `<system-reminder>` 标签，将最新信息传递给模型，从而保护缓存不被破坏。"

通过这种方式传递的信息包括：
- 当前时间和日期变化（"现在是周三下午 3:45"）
- 通过文件修改时间检测到的外部文件变更
- 模式切换（进入/退出计划模式）
- 来自 Hook 或插件的更新上下文

这种模式确保缓存在每一轮交互中保持不变的前缀，在长会话中维持极高的缓存命中率。

---

### 经验三：对话中途不要切换模型

提示词缓存是**模型专属的**——这个事实让模型切换的经济账变得相当反直觉。

假设你已经用 Opus 聊了 10 万 token，此时有一个比较简单的问题要问。直觉上，切成更便宜的 Haiku 来回答应该更省钱吧？**错。切换到 Haiku 反而比让 Opus 继续回答更贵**，因为切换模型意味着必须从头重建整个提示词缓存。

原因在于，KV Cache 是与模型权重绑定的。Opus、Sonnet 和 Haiku 的缓存互相完全隔离。一旦切换模型，下一次请求就必须以零缓存的方式重新读取整段对话历史。

**解决方案：子 Agent。** 如果某个子任务确实需要不同的模型，就派一个子 Agent 去处理。子 Agent 拥有自己独立的上下文和缓存，完成任务后将结果传回，主会话的缓存链路毫发无损。Claude Code 内置的 Explore Agent 正是这样做的——它们以 Haiku 作为子 Agent 运行，避免破坏主会话的缓存。

官方建议很明确：**在会话开始时就选好模型，并且一路用到底。**

---

### 经验四：绝不在会话中途增删工具

工具定义是缓存前缀的一部分。添加或移除一个工具，会导致整个对话的提示词缓存全部失效。这是团队观察到的"最常见的缓存破坏方式之一"。

Claude Code 通过以下设计，在不修改工具集合的前提下实现了状态切换：

**将计划模式做成工具：** 与其在进入计划模式时替换整套工具集（这会破坏缓存），Claude Code 始终保留全部工具，并把计划模式实现为两个工具——`EnterPlanMode` 和 `ExitPlanMode`。当 Agent 进入计划模式时，它会收到一条系统消息，告知其只能探索代码、不能编辑文件，完成规划后调用 `ExitPlanMode`。还有一个额外好处：模型在遇到难题时，可以**自主**调用 `EnterPlanMode` 进入计划模式——不需要用户手动切换。

**MCP 工具的延迟加载（`defer_loading`）：** 对于 MCP（模型上下文协议）工具，Claude Code 不会移除未使用的工具。相反，它发送的是轻量级"桩"——只包含工具名称，标记为 `defer_loading: true`。完整的工具 schema 只有在模型通过 `ToolSearch` 工具主动选择并需要时才加载。由于每次请求中相同的桩始终以相同顺序存在，缓存前缀得以保持稳定。

核心原则：**用工具来建模状态转换，而不是修改工具集本身。**

---

### 经验五：压缩上下文而不破坏缓存

当一个 Claude Code 会话填满上下文窗口时，必须进行压缩（compaction）——将对话历史总结为摘要，以便在新会话中继续。天真的做法是发起一个独立的 API 调用，使用不同于主会话的系统提示词（比如"请总结以下对话"）——这是一个巨大的成本陷阱。因为提示词缓存是前缀匹配的，使用不同的系统提示词意味着**没有任何**已缓存的对话内容能被复用，你将为整个对话历史支付全额的、未经缓存的输入价格。

Claude Code 的解决方案称为**缓存安全的分叉（Cache-Safe Forking）：**

- 压缩请求使用与父会话**完全相同的**系统提示词、用户上下文、系统上下文和工具定义。
- 它将父会话的对话消息作为前缀，然后在末尾以一条新的用户消息的形式追加压缩指令。
- 从 API 的视角来看，这个请求和父会话的最后一次请求几乎一模一样——相同的前缀、相同的工具、相同的历史——因此**缓存前缀被完整复用**。唯一新增的 token 开销就是压缩指令本身。

这需要预留一个"压缩缓冲区"——在上下文窗口中留出足够的空间来容纳压缩指令和摘要输出 token。但节省是巨大的：不需要为可能长达数十万 token 的对话历史支付全价，只需为压缩指令和摘要输出付费。

---

### 经验六：像监控系统可用性一样监控缓存命中率

缓存命中率下降几个百分点，就能对成本和延迟造成显著影响。在长对话场景下，一次缓存未命中（cache miss）不仅仅是"贵一点"——它可能贵上 10 倍，同时慢得多，因为整个前缀都需要重新计算。

Claude Code 团队将缓存命中率视为关键基础设施指标：

> "我们确实在监控提示词缓存的命中率，一旦命中率过低，就会触发告警，甚至会宣布 SEV 级别的事故。"

这种运维姿态——把缓存断裂当作生产事故对待——反映了缓存对产品经济的根本性影响。没有高缓存命中率，订阅模式根本跑不通。

## 经济效益：用数字说话

提示词缓存的经济账非常清晰：

- **缓存读取的成本仅为常规输入 token 的 10%**（以 Sonnet 为例：$0.30/百万 token vs $3.00/百万 token）
- **缓存写入成本为基准输入价格的 1.25 倍**（一次性开销，用于填充缓存）

来自 Claude Code 开发过程的真实数据：
- 团队中的一位工程师在一周内通过缓存节省了超过 **3 亿 token**
- 单日缓存 token 量达到 **9100 万**——实际计费仅约 900 万 token
- 如果没有缓存，同样一天的成本将是 10 倍

团队表示，他们的缓存节省量常常超过实际处理的 token 数量，在某些维度上甚至超出很多。

## 缓存生存时间（TTL）

不同上下文对应不同的缓存有效期：

| 上下文 | 缓存 TTL |
|---|---|
| Claude Code 订阅（系统提示词 + 工具定义） | **1 小时** |
| Claude API（默认） | **5 分钟** |
| 子 Agent | **始终 5 分钟** |

订阅会话的 1 小时 TTL 意味着，只要你持续对话（或在 1 小时内恢复会话），系统提示词和工具定义就始终在缓存中。对话消息的 5 分钟 TTL 则意味着短暂停顿不会破坏缓存，但间隔过长就需要重建。

## 结语

提示词缓存不是一个可以在最后才"加上去"的优化——它是一个从第一天起就塑造 Agent 产品每一层架构的根本约束。从每个 API 请求中内容的排列顺序，到工具的设计和加载方式，到模型的选用策略，再到上下文窗口的管理——缓存的考量渗透到方方面面。

这六条经验——为前缀而布局、用消息而非提示词修改来传递更新、对话中途不切换模型或工具、用安全分叉来压缩上下文、像监控可用性一样监控缓存命中率——划定了一条分界线：一边是经济上可以持续运行的 Agent，另一边则不是。

随着越来越多的团队开始构建长时间运行的 Agent 产品，我们希望这些通过大规模构建 Claude Code 的实战经验凝练出来的模式，能帮助其他人避开同样的深坑。
